# Implementation Plan: Skeleton UI Migration

**Branch**: `002-skeleton-ui-migration` | **Date**: 2026-04-09 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/002-skeleton-ui-migration/spec.md`

---

## Summary

Migrate all existing Choreography UI from a hand-rolled CSS custom-property design system to **Skeleton v4** (`@skeletonlabs/skeleton` + `@skeletonlabs/skeleton-svelte`) on top of **Tailwind CSS v4**, as mandated by the project constitution. The migration is purely presentational — no routes, actions, or data models change. All 17 functional requirements reduce to: (1) install + configure Tailwind v4 + Skeleton v4, (2) rebild the six shared components with Skeleton components, (3) migrate every page file to Tailwind utility classes, (4) eliminate all `<style>` blocks project-wide.

Approach: sequential file-by-file migration driven by the task list generated from this plan. All 41 integration tests must pass throughout. The **Vintage** preset theme is used as-is with `data-theme="vintage"` on `<html>`.

## Implementation Phases

### Phase 1 — Infrastructure (blocker for everything else)

Install packages, configure Tailwind v4 Vite plugin, and set up the global stylesheet. Nothing renders correctly until this phase is complete.

**Files**: `package.json`, `vite.config.ts`, `src/app.css`, `src/app.html`

Key steps:
1. `npm install @skeletonlabs/skeleton @skeletonlabs/skeleton-svelte tailwindcss @tailwindcss/vite @tailwindcss/forms @lucide/svelte`
2. Register `tailwindcss()` **before** `sveltekit()` in `vite.config.ts`
3. Replace `src/app.css` with the canonical Skeleton v4 import order (see research.md §1.3)
4. Add `data-theme="vintage"` to `<html>` in `src/app.html`
5. Add the Windows browser bugfix block to `app.css` (see research.md §2.7)
6. Run `npm test` — all 41 tests must still pass (server logic unchanged)

**Verification**: `npm run dev` renders the app with Vintage theme colours; DevTools confirms `data-theme="vintage"` on `<html>`.

---

### Phase 2 — Shared Components (highest visual impact)

Rebuild the six shared components. These are used across all routes, so migration here produces the biggest visible improvement per file changed.

**Files**: `Header.svelte`, `NavTabs.svelte`, `KidSwitcher.svelte`, `ChoreCard.svelte`, `PrizeCard.svelte`, `LeaderboardRow.svelte`

Key steps:
1. **Header** → `AppBar + AppBar.Toolbar + AppBar.Lead + AppBar.Headline + AppBar.Trail`; sticky top with `z-10`; remove `<style>` block
2. **NavTabs** → `Navigation layout="bar"` + `Navigation.Menu` + `Navigation.TriggerAnchor` for each tab; active state via `preset-filled-primary-500`; remove `<style>` block
3. **KidSwitcher** → custom `<a>` btn row (URL-driven selection; see research.md §2.4); remove `<style>` block
4. **ChoreCard** → `card preset-outlined-surface-200-800 p-4 space-y-2`; add `submitting` state to mark-complete button; remove `<style>` block
5. **PrizeCard** → `card preset-filled-surface-100-900 p-4 space-y-2`; remove `<style>` block
6. **LeaderboardRow** → Tailwind flex-row or table-row layout; remove `<style>` block

**Verification**: All shared components render consistently on a test page; no `var(--color-*)` remain in component markup.

---

### Phase 3 — App Shell + Auth Pages

Migrate the two layouts and two auth pages.

**Files**: `(app)/+layout.svelte`, `(auth)/login/+page.svelte`, `(auth)/signup/+page.svelte`

Key steps:
1. **App layout** → replace `<div class="app-shell">` with `<div class="min-h-dvh flex flex-col">`; `<main class="flex-1 max-w-screen-lg mx-auto w-full px-4 py-6">`; remove `<style>` block
2. **Login page** → `card` wrapper with `preset-filled-surface-100-900`; `input` + `label` utility classes; `btn preset-filled` for submit; `submitting` pattern on form; role-switcher as `btn` toggle row; remove `<style>` block
3. **Signup page** → same form conventions as login; remove `<style>` block

**Verification**: `/login` and `/signup` render with Skeleton form elements; tab through fields confirms `@tailwindcss/forms` normalisation.

---

### Phase 4 — App Route Pages

Migrate the eight app-level page files. These are simpler (no component rebuilds) — mainly class replacement and `<style>` block removal.

**Files**: `chores/+page.svelte`, `prizes/+page.svelte`, `leaderboard/+page.svelte`, `admin/chores/+page.svelte`, `admin/kids/+page.svelte`, `admin/prizes/+page.svelte`, `admin/settings/+page.svelte`, `admin/activity/+page.svelte`

For each file:
1. Remove the `<style>` block entirely
2. Replace every `class="…"` that references a legacy custom class with Tailwind / Skeleton utility classes (see data-model.md §2 for the mapping)
3. Replace any remaining `var(--color-*)`, `var(--space-*)`, etc. with Tailwind equivalents
4. Apply `submitting` pattern to any inline form submit buttons not already in a shared component

**Verification**: `grep -r 'var(--' src/` returns zero results; `grep -r '<style>' src/` returns zero results.

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|-----------|
| Tailwind v4 + Vite plugin incompatibility with Vite 8 | Low | High | Verify in Phase 1 immediately; Skeleton docs confirm `@tailwindcss/vite` is the supported approach for Vite projects |
| Skeleton `Navigation layout="bar"` doesn't support `href` active state detection | Medium | Low | Use manual `isActive()` function (already pattern used in legacy NavTabs) to conditionally apply `preset-filled-primary-500` |
| `SegmentedControl` incompatible with URL-driven kid selection | Resolved | — | Decision made in research: use custom `<a>` btn row instead |
| `use:enhance` submitting state causes test failures | Very Low | Low | Integration tests target server actions, not DOM; `disabled` attribute does not affect form submission in server tests |
| Windows browser styling regression | Low | Medium | Add Windows bugfix CSS block to `app.css` (research.md §2.7) |
| Emoji characters in Header/KidSwitcher render incorrectly | Very Low | Low | Emoji are display-only; no CSS dependency |
